# آلية عمل الوكيل الذكي — توثيق تفصيلي شامل

> **تاريخ التوثيق:** 17 أبريل 2026  
> **المصدر:** فحص مباشر للكود المصدري (chat.routes.ts، sequentialEngine.service.ts، promptBuilder.service.ts، n8n.service.ts)

---

## 1. نظرة عامة على المنظومة

الوكيل الذكي في هذا النظام ليس وكيلاً واحداً بل **منظومة متسلسلة من نموذجين مختلفين** تعمل بالتسلسل لإنتاج output عالي الجودة. النموذجان هما:

- **GPT-4o** (من OpenAI) — للبناء والتوليد والتحسين
- **Gemini 2.5 Pro** (من Google) — للمراجعة والتحقق والتقييم

التواصل مع المستخدم يتم عبر **Server-Sent Events (SSE)** مما يتيح إظهار تقدم كل مرحلة في الوقت الفعلي أثناء المعالجة.

## 2. رحلة الرسالة من البداية حتى الرد النهائي

### الخطوة 0 — استقبال الرسالة والمصادقة

```
المستخدم يرسل رسالة → POST /api/v1/chat/conversations/:id/generate
```

- يتحقق الخادم من **JWT Token** الخاص بالمستخدم (عبر middleware)
- يتحقق من صلاحية `use_chat` للمستخدم
- يُحضر مفاتيح API (OpenAI وGemini) من قاعدة البيانات، مُفككةً التشفير AES-256-CBC
- **إذا لم يوجد مفتاح OpenAI** → يُرسل رسالة خطأ فوراً ويوقف المعالجة

---

### الخطوة 1 — كشف اللغة

```typescript
const isArabic = /[\u0600-\u06FF]/.test(content);
const lang = isArabic ? "ar" : "en";
```

يكشف النظام تلقائياً إذا كانت الرسالة عربية أم إنجليزية، وعلى أساسها يختار:
- البرومبتات المناسبة لكل مرحلة (عربي/إنجليزي)
- لغة الرد النهائي للمستخدم

---

### الخطوة 2 — تحديد النية (Intent Detection)

يُحلل النظام الرسالة لتصنيفها في واحدة من **ثلاث نوايا**:

#### أ) نية الإنشاء (Create Intent)
يبحث عن الكلمات المفتاحية:

| عربي | إنجليزي |
|------|---------|
| أنشئ، اصنع، ابني، إنشاء | create, build, make, generate |
| أريد workflow، أريد سير عمل | new workflow, workflow for, automate |
| أتمتة، سير عمل، أوتوماتيك | i want a workflow, i need a workflow |

#### ب) نية التعديل (Modify Intent)
يبحث عن الكلمات المفتاحية:

| عربي | إنجليزي |
|------|---------|
| عدّل، غيّر، اصلح، حدّث | modify, edit, update, change, fix |
| أضف، احذف، ازل، ادمج | add, remove, delete, adjust, merge |
| بدّل، طوّر، شغّل، وقّف | enable, disable, connect, rename |

#### ج) نية الاستفسار/الدردشة (Query/Chat Intent)
كل ما لا ينطبق عليه التصنيفان أعلاه.

## 3. مسارات المعالجة التفصيلية

### المسار A — إنشاء Workflow جديد (المسار الرئيسي)

> **يُفعَّل عند:** كشف نية الإنشاء + وجود مفتاحَي OpenAI وGemini معاً

هذا هو **المحرك التسلسلي** المكوّن من 4 مراحل:

---

#### 🔵 المرحلة 1A — GPT-4o: تحليل الـ Nodes المطلوبة

**البرومبت (System):**
```
أنت مهندس n8n workflow. مهمتك الوحيدة تحديد أي nodes n8n مطلوبة.
أرجع JSON بهذا الشكل فقط:
{
  "triggerNode": "نوع الـ trigger node",
  "processingNodes": ["نوع1", "نوع2"],
  "outputNodes": ["نوع1"],
  "reasoning": "جملة واحدة تشرح البنية"
}
```

**البرومبت (User):**
```
طلب المستخدم: "[رسالة المستخدم]"
اذكر أنواع الـ n8n nodes المطلوبة فقط. أرجع JSON فقط.
```

**الإعدادات:**
- النموذج: `gpt-4o`
- درجة الحرارة: `0.1` (حتمية جداً)
- أقصى tokens: `500`
- صيغة الرد: `json_object`

**الهدف:** معرفة أنواع الـ nodes الدقيقة قبل البناء (مثل: `n8n-nodes-base.gmail`، `n8n-nodes-base.slack`)

---

#### 🔵 المرحلة 1B — GPT-4o: بناء الـ Workflow JSON

**الميزة الأساسية:** يُحقن في البرومبت **مواصفات دقيقة للـ nodes** من مكتبة داخلية (`nodeSchemas`) بناءً على الـ nodes التي حددتها المرحلة 1A. هذه المواصفات تحتوي على:
- النوع الدقيق (type)
- الإصدار الصحيح (typeVersion)
- اسم الـ credential المطلوب
- الـ parameters المتاحة

**يُضاف أيضاً مثال تطبيقي كامل (Few-Shot Example):**
```json
{
  "nodes": [
    { "id": "uuid", "name": "Gmail Trigger", "type": "n8n-nodes-base.gmailTrigger", 
      "typeVersion": 1, "position": [240, 300], 
      "credentials": { "gmailOAuth2": "gmailOAuth2" }, "parameters": {...} },
    { "id": "uuid", "name": "Send Slack Notification", "type": "n8n-nodes-base.slack",
      "typeVersion": 2, "position": [480, 300], 
      "credentials": { "slackApi": "slackApi" }, "parameters": {...} }
  ],
  "connections": { "Gmail Trigger": { "main": [[{ "node": "Send Slack Notification", "type": "main", "index": 0 }]] } },
  "settings": { "executionOrder": "v1" }
}
```

**الإعدادات:**
- النموذج: `gpt-4o`
- درجة الحرارة: `0.2`
- أقصى tokens: `4000`
- صيغة الرد: `json_object`

**الناتج:** JSON كامل للـ workflow يُخزَّن كـ `phase1Result`

---

#### 🟣 المرحلة 2 — Gemini 2.5 Pro: المراجعة والتقييم

يأخذ Gemini الـ JSON الناتج من المرحلة 1 ويُقيّمه على 7 معايير:

1. **صحة JSON** — هل البنية صحيحة ومتوافقة مع n8n؟
2. **أسماء node types** — هل تبدو صحيحة؟
3. **typeVersion** — هل قيمه منطقية؟
4. **أسماء Credentials** — هل تبدو صحيحة؟
5. **الـ Connections** — هل تستخدم أسماء الـ nodes بالضبط؟
6. **منطق العمل** — هل يُحقق الـ workflow الهدف المطلوب؟
7. **معالجة الأخطاء** — هل يوجد error handling كافٍ؟

**الناتج من Gemini:**
```json
{
  "overallScore": 78,
  "approved": false,
  "criticalIssues": ["مشكلة 1", "مشكلة 2"],
  "improvements": ["تحسين 1"],
  "strengths": ["نقطة إيجابية 1"],
  "specificFixes": ["إصلاح محدد 1"],
  "reviewSummary": "ملخص المراجعة"
}
```

**الإعدادات:**
- النموذج: `gemini-2.5-pro`
- درجة الحرارة: `0.2`
- أقصى tokens: `2000`

> **⚡ اختصار ذكي:** إذا كان `overallScore >= 85` و`approved = true`، تُتخطى المرحلة 3 كلياً وتنتقل مباشرة للمرحلة 4.

#### 🔵 المرحلة 3 — GPT-4o: التحسين بناءً على تقرير Gemini

يُرسَل لـ GPT-4o ثلاثة عناصر:
1. الطلب الأصلي للمستخدم
2. الـ workflow JSON من المرحلة 1
3. تقرير المراجعة الكامل من Gemini (المرحلة 2)

**التعليمات:**
- طبّق جميع الإصلاحات الحرجة (criticalIssues)
- طبّق جميع الإصلاحات المحددة (specificFixes)
- احتفظ بالنقاط الإيجابية (strengths)
- أرجع JSON صالح فقط

**الإعدادات:**
- النموذج: `gpt-4o`
- درجة الحرارة: `0.2`
- أقصى tokens: `4000`

**الناتج:** JSON محسَّن يُخزَّن كـ `phase3Result`

### المسار C — تعديل Workflow موجود

> **يُفعَّل عند:** نية التعديل + وجود مفتاح OpenAI

#### الخطوة C1 — تحديد الـ Workflow المقصود

1. يجلب قائمة جميع الـ workflows من n8n
2. يُرسل لـ GPT-4o: أسماء وIDs الـ workflows + رسالة المستخدم
3. GPT-4o يُحدد أي workflow يقصده المستخدم
4. **إذا لم يُحدَّد الـ workflow** → يُرسل رسالة للمستخدم يطلب التوضيح

#### الخطوة C2 — جلب الـ Workflow الكامل من n8n

```
GET /api/v1/workflows/{id}
```

يجلب الـ JSON الكامل للـ workflow بما فيه جميع الـ nodes والـ connections والإعدادات.

#### الخطوة C3 — توليد التعديل (workflowModifier.service.ts)

يُرسل لـ GPT-4o:
- الـ workflow الحالي (JSON كامل)
- طلب التعديل من المستخدم
- تعليمات صارمة بعدم تغيير ما لم يُطلب

**إذا كان Gemini متاحاً:** يُراجع الـ workflow المعدَّل قبل التطبيق.

#### الخطوة C4 — تطبيق التعديل على n8n

```
PUT /api/v1/workflows/{id}
payload: { name, nodes, connections, settings }
```

**إذا نجح التطبيق:** يُخبر المستخدم أن التعديل طُبِّق مباشرة على n8n  
**إذا فشل:** يُعطيه الـ JSON المعدَّل لاستيراده يدوياً

### المسار D — الدردشة والاستفسار العام

> **يُفعَّل عند:** لا نية إنشاء ولا نية تعديل

**الإعدادات:**

## 4. اتصال n8n بالتفصيل

### آلية الاتصال

```
قاعدة البيانات (مُشفَّر AES-256)
       ↓ فك تشفير
{ url: "https://n8n.example.com", apiKey: "n8n_api_xxx" }
       ↓
HTTP Request → {n8nUrl}/api/v1/{endpoint}
Headers: { "X-N8N-API-KEY": apiKey, "Content-Type": "application/json" }
Timeout: 30 ثانية
```

### العمليات المتاحة على n8n

| العملية | HTTP Method | المسار | الاستخدام |
|---------|------------|--------|----------|
| جلب جميع الـ workflows | GET | `/workflows` | سياق الدردشة + كشف التعديل |
| جلب workflow محدد | GET | `/workflows/{id}` | التعديل + الدردشة التفصيلية |
| تحديث workflow | PUT | `/workflows/{id}` | تطبيق التعديل |
| إنشاء workflow | POST | `/workflows` | الاستيراد المباشر |
| تفعيل workflow | POST | `/workflows/{id}/activate` | من واجهة الـ Workflows |
| إيقاف workflow | POST | `/workflows/{id}/deactivate` | من واجهة الـ Workflows |
| حذف workflow | DELETE | `/workflows/{id}` | من واجهة الـ Workflows |
| جلب سجل التنفيذ | GET | `/executions?workflowId={id}` | تحليل الأخطاء |
| جلب تفاصيل تنفيذ | GET | `/executions/{id}` | تحليل الأخطاء التفصيلي |

### ما تُوفره n8n للوكيل الذكي

1. **قائمة الـ Workflows:** الاسم، ID، حالة التفعيل، عدد الـ nodes
2. **JSON الـ Workflow الكامل:** كل الـ nodes والـ connections والإعدادات
3. **سجل التنفيذ:** آخر 15-20 تنفيذ مع الحالة (نجاح/فشل)
4. **تفاصيل الأخطاء:** رسالة الخطأ، اسم الـ node الذي فشل، الـ stack trace
5. **إصدار n8n** (يُجلب بثلاث طرق مختلفة كـ fallback)

## 5. المحاكاة وبيئة الاختبار

### ماذا يحدث بدون مفتاح OpenAI؟

```json
event: error
data: { "message": "مفتاح OpenAI غير مضبوط. يرجى الذهاب للإعدادات." }
```
→ توقف كامل

### ماذا يحدث بدون مفتاح Gemini؟

- **نية الإنشاء:** يعمل بـ GPT-4o فقط (مسار B) — جودة C
- **نية التعديل:** يعمل بـ GPT-4o فقط بدون مراجعة
- **الدردشة:** يعمل بالكامل (Gemini غير مطلوب للدردشة)

### ماذا يحدث إذا لم يكن n8n مضبوطاً؟

- **نية التعديل:** يُرسل رسالة للمستخدم يطلب التوضيح أو ضبط n8n
- **الدردشة:** يعمل بدون سياق الـ workflows (يُجيب بمعرفته العامة)
- **نية الإنشاء:** يعمل بشكل طبيعي (لا يحتاج n8n لإنشاء JSON)

### الـ Fallbacks التلقائية في المحرك التسلسلي

| حالة الفشل | السلوك |
|-----------|--------|
| فشلت المرحلة 1A (node analysis) | تُتخطى وتذهب مباشرة للمرحلة 1B |
| فشلت المرحلة 2 (Gemini review) | تُستخدم مراجعة افتراضية (score: 75) وتكمل |
| فشلت المرحلة 3 (GPT refine) | يُستخدم الـ JSON من المرحلة 1 بدلاً منه |
| فشلت المرحلة 4 (Gemini validate) | يُستخدم تقرير افتراضي (grade: B) |

```
        ↓
[كشف اللغة: عربي/إنجليزي]
        ↓
[كشف النية]
        ↓
    ┌───┴────────────────────────────────────────┐
    │                    │                      │
نية إنشاء          نية تعديل             دردشة/استفسار
    │                    │                      │
[OpenAI + Gemini؟]    [جلب Workflows     [جلب Workflows
    │                  من n8n]              من n8n]
    ├──كلاهما            │                      │
    │    ↓          [كشف Workflow            [GPT-4o مع
    │ المحرك          المقصود]             سياق n8n +
    │ التسلسلي       [جلب JSON]           سجل المحادثة]

    └───┬───────────────┴──────────────────────┘
        ↓
[حفظ في قاعدة البيانات]
        ↓
[SSE → رد نهائي للمستخدم]
```

## 7. إمكانيات الوكيل — ملخص شامل

### ما يستطيع فعله:

✅ إنشاء n8n workflow كامل من وصف نصي (عربي أو إنجليزي)  
✅ مراجعة وتقييم جودة الـ workflow (A+ وصولاً لـ F)  
✅ تحسين الـ workflow تلقائياً بناءً على نتائج المراجعة  
✅ كشف اسم الـ workflow المقصود من الرسالة  
✅ تعديل workflow موجود في n8n وتطبيق التعديل مباشرة  
✅ الإجابة على أسئلة عامة عن n8n والأتمتة  
✅ شرح محتوى أي workflow موجود في n8n  
✅ تحليل سجلات التنفيذ والأخطاء  
✅ توليد تقرير جودة مفصَّل لكل workflow مُنشَأ  
✅ دعم اللغتين العربية والإنجليزية  

### ما لا يستطيع فعله:

❌ التشغيل الفعلي لـ workflow (فقط تفعيل/إيقاف)  
❌ الوصول لمحتوى الـ executions التاريخية بشكل كامل  
❌ إنشاء credentials في n8n (يُوجِّه المستخدم لإنشائها يدوياً)  
❌ التعامل مع nodes غير الموجودة في مكتبة الـ schemas الداخلية  
❌ الاتصال بأنظمة خارجية أخرى غير n8n وOpenAI وGemini

## 8. معلومات تقنية إضافية للمراجعة

### تفاصيل التخزين في قاعدة البيانات

| الجدول | ما يُخزَّن |
|--------|----------|
| `conversations` | id، userId، عنوان، نوع، حالة، عدد الرسائل |
| `messages` | محتوى الرسالة، الدور (user/assistant)، النموذج المستخدم |
| `generationSessions` | نتائج المراحل الأربع، الـ JSON النهائي، درجة الجودة، الزمن |
| `systemSettings` | مفاتيح API (مُشفَّرة)، URL الـ n8n |

### تدفق SSE (Server-Sent Events)

| الحدث | المحتوى |
|-------|---------|
| `start` | نوع المسار (sequential / gpt-only / modify / chat) |
| `phase` | رقم المرحلة، الاسم، الحالة (pending/running/done/failed)، المدة |
| `complete` | الرسالة النهائية + الـ JSON + التقييمات |
| `error` | رسالة الخطأ |
