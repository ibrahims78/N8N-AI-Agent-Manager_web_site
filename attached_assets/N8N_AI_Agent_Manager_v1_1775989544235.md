# دراسة مقترح تطبيق: N8N AI Agent Manager

---

## نظرة عامة

**N8N AI Agent Manager** تطبيق ويب احترافي متكامل يُتيح التحكم الكامل في n8n عبر محادثة ذكية بالعربية والإنجليزية، دون الحاجة للدخول إلى واجهة n8n مباشرة.

يعتمد التطبيق على نموذجَي **OpenAI GPT-4.1** للإنشاء والتحسين و**Google Gemini 2.5 Pro** للمراجعة والتحقق، يعملان معاً في نظام تحسين تسلسلي رباعي المراحل بحد أقصى 3 جولات مراجعة لإنتاج أعلى جودة ممكنة من الـ workflows.

يتكون التطبيق من **10 صفحات احترافية** تشمل: لوحة التحكم، المحادثة مع الـ Agent، إدارة الـ Workflows، القوالب الجاهزة، سجل المحادثات، وإدارة المستخدمين. يدعم التطبيق **نظام مستخدمين ثنائي المستوى**: مدير بكامل الصلاحيات ومستخدمون بصلاحيات يحددها المدير بدقة. يأتي التطبيق مع **حساب مدير افتراضي** (اسم المستخدم: مدير / كلمة المرور: 123456) يُجبر على تغيير كلمة المرور عند أول دخول.

تدعم الواجهة **اللغة العربية بشكل كامل RTL** مع إمكانية التبديل للإنجليزية، وتتوفر أوضاع **نهاري وليلي** قابلة للتبديل، والافتراضي هو العربية والوضع النهاري.

---

## أولاً: المشكلة التي يحلها التطبيق

### المشاكل التقنية:
- واجهة n8n معقدة تتطلب خبرة تقنية عالية لإنشاء الـ workflows وتعديلها
- إنشاء workflows يدوياً يستهلك وقتاً طويلاً ويُعرّضها لأخطاء بشرية
- لا توجد أداة مركزية للتحكم في n8n باللغة العربية
- صعوبة تشخيص أخطاء الـ workflows وإيجاد الحلول المناسبة بسرعة

### مشاكل إدارة الفريق:
- لا يوجد نظام تحكم بصلاحيات المستخدمين في n8n بشكل مرن
- صعوبة منح فريق العمل وصولاً محدوداً وآمناً إلى n8n
- غياب سجل مرئي لمن أنشأ أو عدّل أي workflow ومتى
- عدم وجود رقابة مركزية على أداء الـ workflows لكل أعضاء الفريق

### مشاكل الجودة والموثوقية:
- النماذج الواحدة تُنتج workflows متوسطة الجودة دون مراجعة
- لا يوجد نظام إصدارات للـ workflows أو إمكانية الرجوع لنسخة سابقة
- غياب ملخص يومي ذكي يشرح أداء الـ workflows والمشاكل المتكررة
- صعوبة تتبع تاريخ التعديلات وتطور الـ workflows عبر الزمن

### مشاكل تجربة المستخدم:
- واجهة n8n الإنجليزية فقط تُصعّب على غير الناطقين بالإنجليزية
- لا يوجد مكان مركزي يجمع الإحصائيات والتنبيهات ونشاط الـ workflows
- إدارة مفاتيح API لنماذج الذكاء الاصطناعي تفتقر للتحقق والمراقبة

## ثانياً: الميزات الرئيسية المقترحة

### 1. نظام التحسين التسلسلي الرباعي (القلب الأساسي)
- **المرحلة 1:** GPT-4.1 يحلل الطلب ويبني النسخة الأولى من الـ workflow
- **المرحلة 2:** Gemini 2.5 Pro يراجع الجودة ويُرسل ملاحظات تفصيلية
- **المرحلة 3:** GPT-4.1 يُطبّق التحسينات بناء على مراجعة Gemini
- **المرحلة 4:** التحقق النهائي من صحة JSON والإرسال المباشر لـ n8n
- بحد أقصى 3 جولات مراجعة لكل workflow لضمان أعلى جودة
- شريط تقدم مرئي يعرض كل مرحلة مع وقتها الفعلي للمستخدم

### 2. واجهة المحادثة الذكية مع الـ Agent
- محادثة طبيعية بالعربي والإنجليزي في نفس الجلسة
- الـ Agent يفهم السياق ويتذكر تاريخ المحادثة كاملاً
- يحول الأوصاف البشرية إلى workflows جاهزة ومُختبرة في n8n
- أسئلة توضيحية ذكية في وضع المبتدئ قبل البناء
- رفع ملفات JSON لتعديلها مباشرة دون كتابة كود
- رفع صور لتشخيص الأخطاء البصرية (GPT-4.1 Vision)
- Streaming للردود (الكتابة التدريجية الحية)
- دعم Drag & Drop لرفع الملفات والصور

### 3. إدارة الـ Workflows الاحترافية
- عرض Card View و List View مع تبديل حر بينهما
- تنظيم الـ workflows في مجلدات مع سحب وإفلات
- شريط نجاح ملوّن حسب نسبة النجاح (أخضر/برتقالي/أحمر)
- مؤشر حي لأي workflow يعمل في الوقت الفعلي
- إجراءات جماعية (Bulk Actions): تفعيل، حذف، تصدير دفعة واحدة
- بحث عميق وفلتر متعدد المعايير
- قائمة سياق لكل workflow: نسخ، نقل، تصدير JSON، عرض السجل

### 4. تفاصيل Workflow مع نظام الإصدارات
- معاينة بصرية حقيقية للـ workflow بالعقد والروابط
- سجل كامل لكل تنفيذ مع الوقت ونتيجة النجاح أو الفشل
- سجل إصدارات تلقائي مع وصف كل تعديل
- مقارنة Diff بصرية بين إصدارين مختلفين
- استعادة فورية لأي إصدار سابق مع نافذة تأكيد
- ضغطة واحدة على أي خطأ تفتح الـ Agent مع بيانات الخطأ

### 5. لوحة التحكم الذكية (Dashboard)
- 4 بطاقات إحصائية رئيسية مع نسب التغيير عن الأمس والأسبوع
- ملخص يومي ذكي من GPT-4.1 يشرح الأداء ويُوصي بالحلول
- رسم بياني تفاعلي للتنفيذات (يومي/أسبوعي/شهري)
- تغذية حية للتنفيذات الأخيرة مع تحديث تلقائي كل 30 ثانية
- مركز تنبيهات مع زر تشخيص مباشر لكل تنبيه
- خريطة حرارة سنوية للنشاط (GitHub-style Heatmap)
- أكثر الـ workflows نشاطاً هذا الأسبوع

### 6. القوالب الجاهزة (Templates)
- مكتبة قوالب جاهزة للاستخدامات الشائعة (إيميل، تقارير، API، جدولة)
- معاينة بصرية للقالب قبل استخدامه
- عند الاختيار يفتح الـ Agent لتخصيص القالب حسب الاحتياج
- عرض عدد العقد وتقدير وقت الإعداد وتقييمات المستخدمين
- حفظ workflows ناجحة كقوالب مخصصة للمستقبل

### 7. سجل المحادثات والتاريخ
- Timeline مرئي لكل محادثة يعرض كل خطوة بالتسلسل
- تصنيف المحادثات: إنشاء / تعديل / تشخيص / استفسار
- بحث داخل محتوى المحادثات وليس فقط عناوينها
- مواصلة أي محادثة قديمة من نقطة توقفها
- إعادة تنفيذ نفس الطلب بضغطة واحدة
- تصدير المحادثة كـ PDF أو JSON
- المدير يرى محادثات جميع المستخدمين

### 8. نظام إدارة المستخدمين (Admin Panel)
- **حساب افتراضي:** اسم المستخدم (مدير) وكلمة المرور (123456) مع إجبار التغيير عند أول دخول
- **نوعان من الحسابات:** مدير (كامل الصلاحيات دائماً) ومستخدم (صلاحيات مخصصة)
- **10 صلاحيات قابلة للتفعيل والإيقاف** لكل مستخدم بشكل مستقل
- إنشاء الحسابات حصرياً من المدير — لا يوجد تسجيل ذاتي
- تفعيل/إيقاف الحساب مؤقتاً بدون حذفه
- إعادة تعيين كلمة المرور مع إشعار تلقائي للمستخدم
- عرض نشاط كل مستخدم (آخر دخول، عدد المحادثات، آخر workflow)
- حماية تلقائية: قفل الحساب بعد 5 محاولات دخول فاشلة

### 9. دعم اللغة العربية والوضع الليلي
- **واجهة عربية كاملة RTL** بخط Tajawal أو IBM Plex Arabic
- **زر تبديل اللغة** (عربي/إنجليزي) في شريط التنقل — **الافتراضي العربي**
- **زر تبديل الوضع** (نهاري/ليلي) في شريط التنقل — **الافتراضي نهاري**
- كلا الاختيارين محفوظان تلقائياً في المتصفح
- الـ Agent يرد بنفس لغة المستخدم (عربي أو إنجليزي)
- الواجهة تستجيب لجميع أحجام الشاشات (جوال، تابلت، كمبيوتر)

### 10. إعدادات متقدمة مع اختبار API
- اختبار مفتاح OpenAI GPT-4.1 بشكل حي مع عرض الرصيد المتبقي
- اختبار مفتاح Gemini 2.5 Pro بشكل حي مع حالة التفعيل
- اختبار كلا المفتاحين معاً مع ملخص شامل لجاهزية النظام التسلسلي
- تحذير تلقائي عند انخفاض رصيد OpenAI مع نصيحة بالشحن
- تاريخ آخر تحديث لكل مفتاح API
- إعداد قواعد التنبيهات وحدود الأداء المقبول
- منطقة خطر محمية بتأكيد مزدوج لأي إجراء حساس

### المرحلة الأولى: استيعاب الطلب
```
المستخدم يكتب طلبه بشكل طبيعي
        ↓
GPT-4.1 يحلل الطلب ويستخلص:
  - الهدف الرئيسي من الـ workflow
  - الـ nodes المطلوبة
  - منطق التدفق (الشروط، الحلقات، الفروع)
  - البيانات المدخلة والمخرجة
        ↓
يبني الـ workflow الأول (النسخة الخام)
```

**ما يظهر للمستخدم:**
```
⏳ GPT-4.1 يحلل طلبك ويبني الـ workflow...
```

---

### المرحلة الثانية: المراجعة النقدية بـ Gemini
```
Gemini يستقبل:
  - طلب المستخدم الأصلي
  - الـ workflow الذي بناه GPT-4.1
        ↓
يراجعه من 6 محاور:

  1. صحة بنية JSON
     هل الـ JSON صالح ومتوافق مع n8n؟

  2. اكتمال الـ Nodes
     هل جميع الـ nodes المطلوبة موجودة؟
     هل هناك nodes زائدة غير ضرورية؟

  3. منطق التدفق (Flow Logic)
     هل الـ connections بين الـ nodes صحيحة؟
     هل الترتيب منطقي؟

  4. معالجة الأخطاء (Error Handling)
     هل يوجد Error node للتعامل مع الفشل؟
     هل كل node يمكن أن يفشل محمي؟

  5. الأداء والكفاءة
     هل يوجد طريقة أبسط لتحقيق نفس الهدف؟
     هل يوجد استدعاءات API غير ضرورية؟

  6. تحقيق هدف المستخدم
     هل الـ workflow يحقق ما طلبه المستخدم فعلاً؟
        ↓
Gemini يُخرج تقرير نقد مفصل بالملاحظات
```

**ما يظهر للمستخدم:**
```
🔍 Gemini يراجع الـ workflow ويبحث عن التحسينات...
```

### نظام حد الجولات (Loop Control)

لمنع الحلقات اللانهائية وضبط التكلفة:

```
الجولة الأولى  ← المراجعة الكاملة
الجولة الثانية ← تحسين ما تبقى
الجولة الثالثة ← آخر فرصة للتحسين
        ↓
بعد 3 جولات: يُرسل أفضل نسخة تم الوصول إليها
              مع إشعار للمستخدم بعدد الجولات
```

**لماذا 3 جولات كحد أقصى؟**
- جولة واحدة: قد لا تكفي للـ workflows المعقدة
- ثلاث جولات: تُنتج جودة ممتازة في 99% من الحالات
- أكثر من ذلك: يرفع التكلفة ويُبطئ التجربة بلا فائدة تُذكر

## خامساً: تدفق العمل الكامل (User Flow)

```
1.  المستخدم يفتح التطبيق
2.  يسجل دخوله (بريد + كلمة مرور)
3.  يضيف بيانات n8n (URL + API Key)
4.  يضيف API Keys (OpenAI + Gemini)
5.  يرى Dashboard بجميع الـ workflows
6.  يكتب في الـ Chat: "أنشئ workflow..."
7.  يبدأ نظام التحسين التسلسلي (4 مراحل)
8.  يرى شريط التقدم أثناء الإنشاء
9.  يستقبل تقرير الجودة النهائي
10. يراجع معاينة الـ workflow
11. يوافق → يُرسل لـ n8n تلقائياً
12. يظهر تأكيد النجاح مع رابط الـ workflow
```

### أولاً: مرحلة الاستيعاب قبل البناء (Pre-Build Interview)

أكثر ما يُضعف دقة الـ workflows المولّدة بالذكاء الاصطناعي هو غموض الطلب الأوّلي. لذلك قبل أن يبدأ النظام بالبناء، يُجري الـ Agent حواراً توضيحياً ذكياً يستخلص منه جميع التفاصيل الضرورية:

```
المستخدم: "أنشئ workflow للإيميل"
        ↓
Agent: قبل البناء، أحتاج توضيح 3 نقاط فقط:
  1. إيميل إرسال أم استقبال ومعالجة؟
  2. هل هو يدوي التشغيل أم مجدول؟
     إذا مجدول ← كل كم؟ (يومي / أسبوعي / عند حدث)
  3. ما خدمة الإيميل: Gmail / Outlook / SMTP مخصص؟
        ↓
المستخدم يجيب → Agent يبني بدقة عالية
```

#### أسئلة يطرحها الـ Agent حسب نوع الطلب

| نوع الطلب | أسئلة الاستيعاب |
|-----------|----------------|
| جلب بيانات API | ما هو رابط الـ API؟ هل يحتاج Authentication؟ ما صيغة البيانات المتوقعة؟ |
| إرسال إشعارات | القناة (Email/Slack/SMS)؟ ما حدث التشغيل؟ ما محتوى الإشعار؟ |
| مزامنة بيانات | المصدر والهدف؟ كيف تُعالَج البيانات المكررة؟ ما معدل المزامنة؟ |
| معالجة ملفات | نوع الملف؟ من أين يأتي؟ ماذا يحدث له بعد المعالجة؟ |

> هذه المرحلة وحدها ترفع دقة الـ workflow الناتج من 60% إلى أكثر من 95%

### ثانياً: نظام Prompt Engineering متخصص لـ n8n

بدلاً من إرسال طلب المستخدم للـ AI كما هو، يُبنى Prompt احترافي ومتخصص يتضمن:

#### مكونات الـ Prompt الاحترافي
```
[1] قواعد بنية n8n JSON الكاملة والدقيقة
[2] قائمة بجميع الـ nodes المتاحة في هذا الإصدار من n8n
[3] الحقول الإجبارية لكل node شائع الاستخدام
[4] أمثلة على workflows صحيحة (Few-Shot Examples)
[5] قواعد Error Handling الإجبارية
[6] قواعد التسمية المنظمة للـ nodes
[7] قيود وتحذيرات الإصدار الحالي من n8n
[8] إجابات المستخدم من مرحلة الاستيعاب
[9] طلب الإخراج بصيغة JSON صارمة فقط
```

#### لماذا هذا مهم؟
النموذج بدون هذا الـ Prompt قد يُنشئ:
- nodes غير موجودة في n8n
- حقولاً خاطئة أو منقوصة
- connections منطقياً غير صحيحة
- JSON غير صالح للاستيراد

### ثالثاً: مكتبة الأمثلة الداخلية (Internal Examples Library)

قبل إرسال الطلب للـ AI، يبحث النظام في مكتبة أمثلة جاهزة ومُختبرة:

```
طلب المستخدم
        ↓
بحث بالتشابه في المكتبة الداخلية
        ↓
وُجد مثال مشابه بنسبة 80%+؟
  نعم ← يستخدمه كنقطة انطلاق ويعدّل عليه
  لا  ← يبني من الصفر بالـ Prompt الكامل
        ↓
النتيجة: دقة أعلى + وقت أسرع + تكلفة أقل
```

#### محتوى المكتبة
- أكثر من 50 workflow مُختبر وجاهز
- مصنّف حسب النوع (إيميل، API، جداول، Webhooks...)
- يتوسع المكتبة تلقائياً بكل workflow ناجح ينشئه المستخدمون

### رابعاً: طبقة التحقق التقني (JSON Validator Layer)

قبل عرض أي workflow للمستخدم، يمر على طبقة تحقق تقنية صارمة ومستقلة:

#### نقاط التحقق الإجبارية

| نقطة التحقق | الوصف |
|------------|-------|
| صحة JSON | هل الـ JSON صالح وكامل بدون أخطاء صياغية؟ |
| اكتمال الـ Nodes | هل كل node لها الحقول الإجبارية؟ |
| منطقية الـ Connections | هل كل node مرتبطة بمصدر ومخرج منطقي؟ |
| وجود Error Handler | هل كل node قد تفشل لها معالج أخطاء؟ |
| صحة الـ Credentials | هل الـ credentials المطلوبة مذكورة بشكل صحيح؟ |
| توافق الإصدار | هل جميع الـ nodes متوافقة مع إصدار n8n المستخدم؟ |

#### ماذا يحدث عند الفشل؟
```
فشل التحقق
        ↓
تحديد نقطة الفشل بدقة
        ↓
إعادة توليد الـ workflow تلقائياً مع تمرير نقطة الفشل للـ AI
        ↓
تحقق ثانٍ → إذا نجح يُعرض / إذا فشل → جولة ثالثة وأخيرة
```

### خامساً: تقسيم الطلبات المعقدة (Task Decomposition)

للـ workflows الكبيرة والمعقدة، بدلاً من بنائها دفعة واحدة (مما يُقلل الدقة)، يقسّمها الـ Agent لوحدات منطقية مستقلة:

```
"أنشئ نظام تقارير يومي كامل يجمع البيانات ويُرسلها"
        ↓
Agent يحلل ويقسّم:
  الوحدة 1: جلب البيانات من المصدر (HTTP / Database node)
  الوحدة 2: معالجة وتصفية البيانات (Function node)
  الوحدة 3: تنسيق التقرير (Set node)
  الوحدة 4: إرسال التقرير (Gmail node)
  الوحدة 5: Error Handler شامل لكل وحدة
        ↓
يبني كل وحدة بدقة عالية
        ↓
يدمج الوحدات في workflow متكامل واحد
        ↓
يمر على نظام المراجعة التسلسلي (GPT-4.1 + Gemini)
```

#### متى يُفعَّل التقسيم؟
- إذا تجاوز الطلب 5 خطوات منطقية
- إذا شمل أكثر من مصدر بيانات واحد
- إذا كان يحتوي على شروط متفرعة (If / Switch)

### أولاً: الجلب الذكي قبل التعديل (Smart Fetch)

أول خطوة قبل أي تعديل هي جلب الـ workflow الحالي بالكامل وفهم بنيته، بدلاً من التعديل الأعمى:

```
المستخدم: "عدّل على workflow الإيميل وأضف تأخير ساعة"
        ↓
Agent يجلب الـ workflow الحالي من n8n بالكامل
        ↓
GPT-4.1 يحلل البنية الكاملة:
  - عدد الـ nodes وأنواعها
  - ترتيب الـ connections
  - الإعدادات والـ credentials الموجودة
  - نقاط الـ Error Handling الحالية
        ↓
يحدد بدقة أين ومتى يُجرى التعديل
        ↓
يُعدّل بذكاء بدون المساس بباقي الـ workflow
```

#### لماذا هذا ضروري؟
بدون الجلب الذكي، قد يُعيد الـ AI بناء الـ workflow من الصفر مما يؤدي إلى:
- فقدان الـ credentials المُعدّة
- فقدان الإعدادات الدقيقة لكل node
- تغيير ما لم يطلب المستخدم تغييره
- خسارة تاريخ التنفيذ المرتبط

### ثانياً: التعديل الجزئي الآمن (Surgical Edit)

المبدأ الأساسي: **لا تلمس ما لا يحتاج للتعديل**

```
الـ workflow يحتوي 8 nodes
المستخدم يطلب تعديل node واحدة فقط
        ↓
النظام يحدد الـ nodes المتأثرة بالتعديل
  (قد تكون 1 أو 2 أو 3 nodes مترابطة)
        ↓
يُعدّل تلك الـ nodes حصراً
        ↓
يُبقي الـ 5 nodes الأخرى كما هي تماماً
        ↓
يُعيد بناء الـ connections بدقة
```

#### ما يُحافَظ عليه دائماً
- جميع الـ credentials والمصادقات
- إعدادات الـ nodes غير المتأثرة
- منطق Error Handling الموجود
- الـ metadata والوصف والاسم
- تاريخ تنفيذ الـ workflow

### رابعاً: التعديل بالأوامر الطبيعية (Natural Language Edits)

الـ Agent يفهم ويُنفّذ أوامر التعديل المكتوبة بلغة طبيعية:

| الأمر الطبيعي | ما يفعله النظام بدقة |
|--------------|---------------------|

| "أضف رسالة تأكيد بعد الإرسال" | يُضيف node في المكان الصحيح ويُعيد الربط |
| "احذف خطوة المعالجة" | يحذف الـ node ويُصلح الـ connections تلقائياً |
| "اجعله يعمل كل ساعة بدل كل يوم" | يعدّل إعداد الجدولة فقط |
| "أضف Error Handler لكل الـ nodes" | يُضيف معالج أخطاء شامل دون المساس بالمنطق |
| "غيّر عنوان الإيميل المُرسَل إليه" | يعدّل حقل واحد في Gmail node فقط |
| "ضاعف التأخير الحالي" | يقرأ قيمة التأخير الحالية ويضاعفها |
| "انسخ هذا الـ workflow وعدّل عليه" | يُنشئ نسخة جديدة ثم يُعدّل عليها |

- البحث فوري يظهر النتائج أثناء الكتابة
- تمييز الكلمة المبحوث عنها بلون مختلف في النتائج

### سادساً: إعادة استخدام المحادثات السابقة

#### أ. إعادة الطلب (Re-run)
```
المستخدم يختار محادثة قديمة ويضغط "إعادة الطلب"
        ↓
نفس الطلب يُرسل مرة أخرى للـ Agent
        ↓
يُبني workflow جديد بأحدث إصدار من النظام
        ↓
مقارنة تلقائية بين الإصدار القديم والجديد
```

#### ب. مواصلة المحادثة (Continue)
```
المستخدم يفتح محادثة قديمة ويضيف طلباً جديداً
        ↓
الـ Agent يتذكر السياق الكامل للمحادثة القديمة
        ↓
يُكمل من حيث توقف بدقة
```

#### ج. نسخ الـ Workflow للـ n8n مجدداً
```
المستخدم يختار محادثة قديمة ويضغط "إرسال لـ n8n"
        ↓
يُرسل الـ workflow الناتج منها مرة أخرى
        ↓
مفيد عند حذف workflow بالخطأ أو نقله لحساب آخر
```

### أولاً: عرض الـ Workflows بطاقات بصرية (Card View)

بدلاً من عرض الـ workflows كجدول مملّ، يُعرض كل workflow ببطاقة بصرية تحتوي على جميع المعلومات الجوهرية دفعة واحدة:

```
┌──────────────────────┐  ┌──────────────────────┐
│ 📧 إيميل يومي        │  │ 📊 تقرير أسبوعي       │
│ ───────────────────  │  │ ───────────────────  │
│ ✅ مفعّل             │  │ ⏸️ موقوف              │
│ آخر تشغيل: منذ 2س   │  │ آخر تشغيل: منذ 3 أيام│
│ نسبة النجاح: 98%    │  │ نسبة النجاح: 75%     │
│ ████████████░░ 98%  │  │ █████████░░░░░ 75%   │
│ ───────────────────  │  │ ───────────────────  │
│ [▶ تشغيل] [✏️] [🗑️] │  │ [▶ تشغيل] [✏️] [🗑️] │
└──────────────────────┘  └──────────────────────┘
```

#### مميزات العرض البصري
- تبديل حر بين Card View و List View حسب تفضيل المستخدم
- ترميز لوني واضح لكل حالة:
  - أخضر ← مفعّل ويعمل بنجاح
  - رمادي ← موقوف
  - أحمر ← مفعّل لكن به أخطاء متكررة
- شريط بياني مصغّر لنسبة النجاح داخل كل بطاقة

### ثانياً: إحصائيات مباشرة على كل Workflow

كل workflow يعرض إحصائياته الحية بدون الحاجة للدخول عليه:

| المعلومة | التفاصيل |
|----------|----------|
| نسبة النجاح | نسبة مئوية مع شريط بياني مصغّر |
| عدد التشغيلات | عدد مرات التشغيل هذا الأسبوع |
| متوسط وقت التنفيذ | بالثواني لكل تشغيل |
| آخر خطأ | رسالة الخطأ مع رابط مباشر للتشخيص |
| التشغيل القادم | للـ workflows المجدولة (Scheduled) |

### ثالثاً: نظام التنظيم المتقدم

#### أ. المجلدات (Folders)
تجميع الـ workflows في مجلدات منطقية مثل:
- 📁 تقارير
- 📁 إيميلات
- 📁 مزامنة البيانات
- 📁 تنبيهات
- 📁 أرشيف

#### ب. التصنيفات (Tags)
إضافة وسوم لكل workflow للبحث والفلترة السريعة:
```
[إيميل] [يومي] [تقارير] [Google Sheets] [مهم]
```

#### ج. خيارات الترتيب الذكي
- الأكثر استخداماً
- الأحدث تعديلاً
- الأعلى نسبة فشل (لمتابعة المشاكل)
- الأبطأ تنفيذاً

#### د. البحث الفوري العميق
- بحث في اسم الـ workflow
- بحث في وصفه
- بحث داخل أسماء الـ nodes في الـ workflow نفسه
- بحث بالـ Tags والمجلدات

```
☑️ إيميل يومي       ☑️ تقرير أسبوعي     ☐ مزامنة Sheets
        ↓
[ ✅ تفعيل الكل ] [ ⏸️ إيقاف الكل ] [ 📦 تصدير ] [ 🗑️ حذف ]
```

|---------|-------|
| تفعيل الكل | تشغيل جميع المحددة مرة واحدة |

| نسخ | استنساخ workflow بنقرة واحدة مع اسم جديد |
| تصدير | تنزيل عدة workflows كملفات JSON |

```

المنفذ الخارجي: لا يُكشف إطلاقاً — داخلي بالكامل
```

**الاستخدامات:**
- **Job Queue:** قائمة انتظار طلبات إنشاء الـ workflows (Bull Queue)
- **Session Cache:** تخزين مؤقت لبيانات الجلسات النشطة

- **JWT Blacklist:** قائمة الـ tokens الملغاة
- **WebSocket Pub/Sub:** توزيع أحداث الـ WebSocket على عدة processes

**الإعداد الموصى به:**
```
maxmemory 256mb
maxmemory-policy allkeys-lru
requirepass <كلمة مرور Redis قوية>
```

### خامساً: إعداد Cloudflare بالتفصيل

#### أ. إعداد الـ DNS Records

| النوع | الاسم | القيمة | Proxy Status |
|-------|-------|--------|-------------|
| A | `@` | IP الـ VPS | 🟠 Proxied |
| A | `app` | IP الـ VPS | 🟠 Proxied |
| A | `n8n` | IP الـ VPS | 🟠 Proxied |
| A | `npm` | IP الـ VPS | 🔘 DNS Only (للوصول الإداري فقط) |

**لماذا Proxied وليس DNS Only؟**
- إخفاء IP الـ VPS الحقيقي عن المهاجمين
- حماية DDoS تلقائية من Cloudflare
- تسريع التحميل عبر Cloudflare CDN
- SSL مجاني وتلقائي بدون تكلفة

#### ب. إعدادات SSL في Cloudflare

```
SSL/TLS Mode: Full (Strict)  ← إجباري وليس Flexible
```

**لماذا Full (Strict) وليس Flexible؟**
- Flexible: Cloudflare ↔ VPS بدون تشفير (خطير!)
- Full (Strict): تشفير كامل في جميع المراحل
- يتطلب شهادة SSL صالحة على الـ VPS (يوفرها NPM)

#### ج. Cloudflare WAF Rules المقترحة

```
Rule 1: Block Access to Admin Paths from Non-Whitelisted IPs
  - Path: /api/admin/*
  - IP Not In: [قائمة IPs المسموح بها للمدير]
  - Action: Block

Rule 2: Rate Limit API Endpoints
  - Path: /api/auth/login
  - Limit: 10 requests per minute per IP
  - Action: Block for 10 minutes

Rule 3: Block Common Attack Patterns
  - Expression: (http.request.uri contains "../../") or 
                (http.request.uri contains "eval(") or
                (http.request.uri contains "<script")
  - Action: Block

Rule 4: Country Block (اختياري حسب الاحتياج)
  - Block requests from high-risk countries if not needed
  - Action: Challenge (CAPTCHA)
```

#### د. Cloudflare Security Settings

```
Security Level:     Medium (يمكن رفعه لـ High للمواقع الحساسة)
Bot Fight Mode:     ON (مُفعّل)
Browser Integrity:  ON (مُفعّل)
Hotlink Protection: ON (منع سرقة الملفات)
Email Obfuscation: ON
```

#### هـ. Cloudflare Tunnel (الخيار الأكثر أماناً)

بدلاً من فتح منافذ على الـ VPS مباشرة، يمكن استخدام **Cloudflare Tunnel**:

```
cloudflared tunnel --url http://localhost:80
```

**مزايا Cloudflare Tunnel:**
- الـ VPS لا يحتاج فتح أي منفذ خارجي (لا 80، لا 443)
- الاتصال من الـ VPS إلى Cloudflare وليس العكس
- حماية تلقائية ضد IP scanning وDDoS
- لا حاجة لإعداد SSL على الـ VPS (Cloudflare يتولاها)
- النشر يعمل حتى خلف NAT أو Firewall صارم

**إعداد Cloudflare Tunnel في Docker Compose:**
```yaml
cloudflare-tunnel:
  image: cloudflare/cloudflared:latest
  container_name: n8n_ai_tunnel
  restart: unless-stopped
  networks:
    - proxy
  command: tunnel --no-autoupdate run --token ${CLOUDFLARE_TUNNEL_TOKEN}
  environment:
    CLOUDFLARE_TUNNEL_TOKEN: ${CLOUDFLARE_TUNNEL_TOKEN}
```

### ثامناً: استراتيجية النسخ الاحتياطي

#### أ. النسخ الاحتياطي التلقائي للـ PostgreSQL

**سكريبت النسخ الاحتياطي اليومي** (`/scripts/backup.sh`):
```bash
#!/bin/bash
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR=/backups
RETENTION_DAYS=30

# نسخ احتياطي لقاعدة بيانات التطبيق
docker exec n8n_ai_postgres pg_dump \
  -U postgres \
  -d n8n_ai_manager \
  --no-password \
  | gzip > "$BACKUP_DIR/app_$DATE.sql.gz"

# نسخ احتياطي لقاعدة بيانات n8n
docker exec n8n_ai_postgres pg_dump \
  -U postgres \
  -d n8n \
  --no-password \
  | gzip > "$BACKUP_DIR/n8n_$DATE.sql.gz"

# نسخ احتياطي لبيانات n8n (credentials وإعدادات)
docker run --rm \
  --volumes-from n8n_ai_n8n \
  -v "$BACKUP_DIR:/backup" \
  alpine tar czf /backup/n8n_data_$DATE.tar.gz /home/node/.n8n

# حذف النسخ القديمة (أقدم من 30 يوم)
find "$BACKUP_DIR" -name "*.gz" -mtime +$RETENTION_DAYS -delete

echo "✅ النسخ الاحتياطي اكتمل: $DATE"
```

**إضافة للـ Crontab:**

```sql
-- دالة تحديث updated_at تلقائياً
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ language 'plpgsql';

-- دالة احتساب نسبة النجاح
CREATE OR REPLACE FUNCTION calculate_success_rate(p_workflow_id UUID)
RETURNS DECIMAL AS $$
DECLARE
    v_total INTEGER;
    v_success INTEGER;
BEGIN
    SELECT COUNT(*), COUNT(*) FILTER (WHERE success = true)
    INTO v_total, v_success
    FROM (
        SELECT (metadata->>'success')::boolean as success
        FROM messages
        WHERE metadata->>'workflow_id' = p_workflow_id::text
          AND created_at > NOW() - INTERVAL '7 days'
    ) executions;
    
    IF v_total = 0 THEN RETURN NULL; END IF;
    RETURN ROUND((v_success::DECIMAL / v_total) * 100, 2);
END;
$$ LANGUAGE plpgsql;
```

### أولاً: تشفير كلمات المرور (bcrypt)

**التطبيق يستخدم bcrypt مع cost factor 12:**

```javascript
const bcrypt = require('bcrypt');
const COST_FACTOR = 12;  // وليس 10 الافتراضي

// عند إنشاء/تغيير كلمة المرور
async function hashPassword(plainPassword) {
    const salt = await bcrypt.genSalt(COST_FACTOR);
    return await bcrypt.hash(plainPassword, salt);
}

// عند التحقق من كلمة المرور
async function verifyPassword(plainPassword, hashedPassword) {
    return await bcrypt.compare(plainPassword, hashedPassword);
}
```

**لماذا cost factor 12 وليس 10؟**
- Cost 10: ~100ms للتحقق ← جيد للتطوير
- Cost 12: ~400ms للتحقق ← مناسب للإنتاج (يُعيق brute-force)
- Cost 14: ~1600ms ← مبالغة تؤثر على تجربة المستخدم

**قواعد التحقق من كلمة المرور الجديدة:**
```
الحد الأدنى: 8 أحرف
يجب احتواء: حرف كبير + حرف صغير + رقم
يُوصى به: رمز خاص (!@#$%)
غير مسموح: كلمات مرور شائعة (قائمة سوداء من 10,000 كلمة)
غير مسموح: كلمة المرور السابقة (مقارنة bcrypt)
```

### ثانياً: نظام JWT المزدوج (Access + Refresh Token)

```
المستخدم يسجل دخولاً
        ↓
الـ Server يُصدر:
  ┌──────────────────────────────────────────┐
  │  Access Token                            │
  │  • مدة الصلاحية: 15 دقيقة               │
  │  • يُرسل في كل طلب API (Authorization Header)  │
  │  • يحتوي: {userId, role, permissions, exp} │
  └──────────────────────────────────────────┘
  ┌──────────────────────────────────────────┐
  │  Refresh Token                           │
  │  • مدة الصلاحية: 7 أيام                 │
  │  • مخزّن في HttpOnly Cookie (لا JavaScript) │
  │  • يُستخدم لتجديد Access Token فقط        │
  │  • مخزّن في قاعدة البيانات للتحقق منه     │
  └──────────────────────────────────────────┘
```

**لماذا هذا التصميم أكثر أماناً؟**

| المشكلة | الحل |
|---------|------|
| سرقة Access Token | انتهاء صلاحيته بعد 15 دقيقة فقط |
| سرقة Refresh Token | HttpOnly Cookie لا يمكن لـ JavaScript قراءته |
| اختراق قاعدة البيانات | الـ Refresh Tokens مشفرة في قاعدة البيانات |
| تسجيل خروج من جهاز واحد | إلغاء Refresh Token المحدد فقط |
| تسجيل خروج من كل الأجهزة | إلغاء جميع Refresh Tokens للمستخدم |

**كود إنشاء Access Token:**
```javascript
const jwt = require('jsonwebtoken');

function generateAccessToken(user, permissions) {
    return jwt.sign(
        {
            userId: user.id,
            role: user.role,
            permissions: permissions
        },
        process.env.JWT_ACCESS_SECRET,
        { expiresIn: '15m', algorithm: 'HS256' }
    );
}

function generateRefreshToken(userId) {
    const token = jwt.sign(
        { userId },
        process.env.JWT_REFRESH_SECRET,
        { expiresIn: '7d', algorithm: 'HS256' }
    );
    return token;
}
```

**تجديد Access Token تلقائياً (Silent Refresh):**
```
Access Token انتهت صلاحيته
        ↓
الـ Frontend يُرسل طلب تجديد (بالـ Refresh Token من Cookie)
        ↓
الـ Backend يتحقق من الـ Refresh Token:
  هل موجود في قاعدة البيانات؟
  هل غير مُلغى؟
  هل لم تنتهِ صلاحيته؟
        ↓
يُصدر Access Token جديد بشكل شفاف
المستخدم لا يشعر بأي انقطاع
```

### ثالثاً: تشفير مفاتيح API (AES-256-GCM)

```javascript
const crypto = require('crypto');

const ALGORITHM = 'aes-256-gcm';
const KEY_LENGTH = 32;  // 256 bit
const IV_LENGTH = 12;   // 96 bit (مثالي لـ GCM)
const TAG_LENGTH = 16;  // 128 bit authentication tag

// مفتاح التشفير من متغيرات البيئة (وليس من قاعدة البيانات)
const MASTER_KEY = Buffer.from(process.env.ENCRYPTION_KEY, 'hex');

function encryptApiKey(plainKey) {
    // IV عشوائي مختلف لكل عملية تشفير
    const iv = crypto.randomBytes(IV_LENGTH);
    
    const cipher = crypto.createCipheriv(ALGORITHM, MASTER_KEY, iv);
    
    const encrypted = Buffer.concat([
        cipher.update(plainKey, 'utf8'),
        cipher.final()
    ]);
    
    const tag = cipher.getAuthTag();
    
    return {
        encryptedKey: Buffer.concat([encrypted, tag]).toString('base64'),
        iv: iv.toString('hex')
    };
}

function decryptApiKey(encryptedKey, ivHex) {
    const iv = Buffer.from(ivHex, 'hex');
    const data = Buffer.from(encryptedKey, 'base64');
    
    // فصل البيانات المشفرة عن الـ authentication tag
    const encrypted = data.slice(0, data.length - TAG_LENGTH);
    const tag = data.slice(data.length - TAG_LENGTH);
    
    const decipher = crypto.createDecipheriv(ALGORITHM, MASTER_KEY, iv);
    decipher.setAuthTag(tag);
    
    return Buffer.concat([
        decipher.update(encrypted),
        decipher.final()
    ]).toString('utf8');
}
```

**لماذا AES-256-GCM وليس AES-256-CBC؟**
- GCM يوفر **Authenticated Encryption** — يكتشف أي تعديل على البيانات المشفرة
- CBC لا يوفر هذه الحماية — قابل لهجمات Padding Oracle
- GCM أسرع على المعالجات الحديثة

### سادساً: حماية قفل الحساب (Account Lockout)

```javascript
const LOCKOUT_CONFIG = {
    max_attempts: 5,           // أقصى محاولات فاشلة
    lockout_duration_minutes: 30,  // مدة القفل
    attempt_window_minutes: 15,    // نافذة زمنية للاحتساب
};

async function handleLoginAttempt(username, password, ip) {
    // تسجيل المحاولة في قاعدة البيانات
    await recordLoginAttempt(username, ip, false);
    
    // احتساب المحاولات الفاشلة في النافذة الزمنية
    const recentFailures = await countRecentFailures(
        username, 
        LOCKOUT_CONFIG.attempt_window_minutes
    );
    
    if (recentFailures >= LOCKOUT_CONFIG.max_attempts) {
        // قفل الحساب
        const lockUntil = new Date(Date.now() + LOCKOUT_CONFIG.lockout_duration_minutes * 60000);
        await lockUserAccount(username, lockUntil);
        
        // تسجيل في audit_log
        await auditLog('account_locked', { username, ip, reason: 'max_attempts_exceeded' });
        
        return { 
            locked: true, 
            message: 'تم قفل حسابك مؤقتاً بسبب محاولات دخول متعددة فاشلة. يرجى التواصل مع مدير النظام أو الانتظار 30 دقيقة.'
        };
    }
    
    const remaining = LOCKOUT_CONFIG.max_attempts - recentFailures;
    return { 
        locked: false, 
        attemptsRemaining: remaining,
        message: `بيانات خاطئة. محاولات متبقية: ${remaining}`
    };
}
```

### ثانياً: التوافق بين إصدارات n8n

#### فحص الإصدار عند الاتصال الأول

```javascript
async function detectN8nVersion(baseUrl, apiKey) {
    // جرب الـ API v1 أولاً
    try {
        const response = await axios.get(`${baseUrl}/api/v1/workflows`, {
            headers: { 'X-N8N-API-KEY': apiKey },
            params: { limit: 1 }
        });
        return { version: 'v1', apiBase: '/api/v1' };
    } catch (error) {
        if (error.response?.status === 404) {
            // قد يكون إصدار قديم يستخدم /api/v0 أو /rest
            try {
                const response = await axios.get(`${baseUrl}/rest/workflows`, {
                    headers: { 'X-N8N-API-KEY': apiKey }
                });
                return { version: 'legacy', apiBase: '/rest' };
            } catch (e) {
                throw new Error('N8N_API_NOT_ACCESSIBLE');
            }
        }
        throw error;
    }
}
```

#### طبقة الـ Adapter لتوحيد الـ API

```javascript
class N8nAdapter {
    constructor(version, baseUrl, apiKey) {
        this.version = version;
        this.baseUrl = baseUrl;
        this.apiKey = apiKey;
    }

    async getWorkflows(options = {}) {
        if (this.version === 'v1') {
            return this._getWorkflowsV1(options);
        } else {
            return this._getWorkflowsLegacy(options);
        }
    }

    async _getWorkflowsV1(options) {
        const response = await this.client.get('/api/v1/workflows', {
            params: { limit: options.limit || 100 }
        });
        // بنية الاستجابة في v1
        return response.data.data.map(wf => this._normalizeWorkflow(wf, 'v1'));
    }

    async _getWorkflowsLegacy(options) {
        const response = await this.client.get('/rest/workflows');
        // بنية الاستجابة في الإصدارات القديمة
        return response.data.data.map(wf => this._normalizeWorkflow(wf, 'legacy'));
    }

    // تطبيع الـ workflow للصيغة الداخلية الموحدة
    _normalizeWorkflow(rawWorkflow, version) {
        if (version === 'v1') {
            return {
                id: rawWorkflow.id,
                name: rawWorkflow.name,
                isActive: rawWorkflow.active,
                nodes: rawWorkflow.nodes,
                connections: rawWorkflow.connections,
                createdAt: rawWorkflow.createdAt,
                updatedAt: rawWorkflow.updatedAt,
                tags: rawWorkflow.tags?.map(t => t.name) || []
            };
        } else {
            return {
                id: String(rawWorkflow.id),
                name: rawWorkflow.name,
                isActive: rawWorkflow.active,
                nodes: rawWorkflow.nodes,
                connections: rawWorkflow.connections,
                createdAt: rawWorkflow.createdAt,
                updatedAt: rawWorkflow.updatedAt,
                tags: rawWorkflow.tags || []
            };
        }
    }
}
```

### خامساً: معالجة أخطاء n8n API

```javascript
class N8nApiError extends Error {
    constructor(statusCode, n8nErrorCode, message) {
        super(message);
        this.statusCode = statusCode;
        this.n8nErrorCode = n8nErrorCode;
    }
}

function handleN8nError(error) {
    if (!error.response) {
        throw new Error('N8N_CONNECTION_FAILED');
    }

    const { status, data } = error.response;

    switch (status) {
        case 400:
            throw new N8nApiError(400, 'INVALID_WORKFLOW', 
                'الـ workflow JSON غير صالح: ' + data.message);
        case 401:
            throw new N8nApiError(401, 'INVALID_API_KEY',
                'مفتاح n8n API غير صحيح أو منتهي الصلاحية');
        case 403:
            throw new N8nApiError(403, 'INSUFFICIENT_N8N_PERMISSIONS',
                'مفتاح API لا يملك الصلاحية اللازمة في n8n');
        case 404:
            throw new N8nApiError(404, 'WORKFLOW_NOT_FOUND',
                'الـ workflow غير موجود في n8n');
        case 409:
            throw new N8nApiError(409, 'WORKFLOW_NAME_CONFLICT',
                'يوجد workflow بنفس الاسم في n8n');
        case 429:
            throw new N8nApiError(429, 'N8N_RATE_LIMIT',
                'تجاوزت حد طلبات n8n API');
        case 500:
        case 502:
        case 503:
            throw new N8nApiError(status, 'N8N_SERVER_ERROR',
                'خطأ داخلي في n8n — قد يكون N8N يُعيد التشغيل');
        default:
            throw new N8nApiError(status, 'UNKNOWN_N8N_ERROR', data.message);
    }
}
```

### سادساً: مزامنة الـ Workflows (Sync Strategy)

```javascript
class WorkflowSyncService {
    // مزامنة كاملة كل 5 دقائق
    async fullSync(userId) {
        const n8nWorkflows = await this.n8nClient.getWorkflows();
        const localCache = await db.query('SELECT * FROM workflow_cache WHERE user_id = $1', [userId]);
        
        const n8nIds = new Set(n8nWorkflows.map(w => w.id));
        const localIds = new Set(localCache.map(w => w.n8n_workflow_id));
        
        // workflows جديدة في n8n
        const toAdd = n8nWorkflows.filter(w => !localIds.has(w.id));
        // workflows محذوفة من n8n
        const toRemove = localCache.filter(w => !n8nIds.has(w.n8n_workflow_id));
        // workflows مُحدَّثة
        const toUpdate = n8nWorkflows.filter(w => localIds.has(w.id));
        
        await Promise.all([
            ...toAdd.map(w => this.addToCache(userId, w)),
            ...toRemove.map(w => this.removeFromCache(w.id)),
            ...toUpdate.map(w => this.updateCache(userId, w))
        ]);
        
        return { added: toAdd.length, removed: toRemove.length, updated: toUpdate.length };
    }
    
    // مزامنة جزئية عند الحاجة (Webhook أو Event)
    async partialSync(userId, workflowId) {
        const n8nWorkflow = await this.n8nClient.getWorkflow(workflowId);
        await this.updateCache(userId, n8nWorkflow);
    }
}
```

## سادس وعشرون: نظام قائمة الانتظار (Queue System)

---

### لماذا نحتاج Queue System؟

```
سيناريو المشكلة:
10 مستخدمين يطلبون إنشاء workflows في نفس اللحظة
        ↓
كل طلب = 4-6 استدعاءات AI (GPT + Gemini)
        ↓
10 × 5 = 50 استدعاء API في نفس الثانية
        ↓
Rate Limit من OpenAI: فشل جماعي
        ↓
نتيجة: أخطاء لجميع المستخدمين
```

**الحل: Bull Queue المبني على Redis**

```
الطلبات تدخل Queue
        ↓
يُعالَج طلب واحد (أو اثنان) في كل مرة
        ↓
نتائج ثابتة بدون أخطاء Rate Limit
```

---

### معمارية الـ Queue System

```javascript
const Queue = require('bull');
const redis = require('ioredis');

const redisClient = new redis(process.env.REDIS_URL);

// Queue لطلبات AI
const aiQueue = new Queue('ai-workflow-generation', {
    redis: { client: redisClient },
    defaultJobOptions: {
        timeout: 300000,      // 5 دقائق timeout
        attempts: 2,          // محاولتان في حال الفشل
        backoff: { type: 'exponential', delay: 5000 },
        removeOnComplete: { count: 100 },
        removeOnFail: { count: 50 }
    }
});

// إعداد المعالجة المتزامنة
// concurrency = 3: ثلاثة طلبات AI بالتوازي
aiQueue.process('create-workflow', 3, async (job) => {
    const { userId, conversationId, userRequest, mode } = job.data;
    
    await job.progress(10);  // تحديث التقدم
    const workflow = await gpt41BuildWorkflow(userRequest);
    
    await job.progress(40);
    const review = await geminiReviewWorkflow(workflow);
    
    await job.progress(70);
    const improved = await gpt41ImproveWorkflow(workflow, review);
    
    await job.progress(90);
    const validated = await geminiValidateWorkflow(improved);
    
    await job.progress(100);
    return { workflow: validated, conversationId };
});
```

### استراتيجية 1: Sliding Window (افتراضية)

```javascript
const CONTEXT_CONFIG = {
    max_messages: 20,           // احتفظ بآخر 20 رسالة
    max_tokens_per_message: 2000, // قصّر الرسائل الطويلة
    always_include_system: true,  // Prompt النظام دائماً موجود
    always_include_first: 2,      // أول رسالتين (سياق البداية)
};

function buildContextWindow(messages) {
    const systemMessages = messages.filter(m => m.role === 'system');
    const firstMessages = messages.slice(0, CONTEXT_CONFIG.always_include_first);
    const recentMessages = messages.slice(-CONTEXT_CONFIG.max_messages);
    
    // دمج بدون تكرار
    const contextMessages = [
        ...systemMessages,
        ...firstMessages,
        ...recentMessages.filter(m => !firstMessages.includes(m))
    ];
    
    return contextMessages;
}
```

### ثانياً: إدارة الحاويات بصرياً (Portainer)

#### ما هو Portainer؟

Portainer هي لوحة تحكم بصرية كاملة لـ Docker تعمل داخل حاوية مستقلة وتتصل بـ Docker socket على الـ VPS. تُتيح إدارة جميع الحاويات بالكامل من المتصفح بدون الحاجة لأوامر SSH في أغلب الأحيان.

#### ما الذي يمكنك فعله من Portainer؟

| الوظيفة | الوصف |
|---------|-------|
| عرض حالة الحاويات | ترى جميع الحاويات الست: تشغيل / إيقاف / إعادة تشغيل / حذف |
| Logs حية | عرض logs أي حاوية في الوقت الفعلي بدون SSH |
| استهلاك الموارد | CPU، RAM، Network، Disk I/O لكل حاوية بشكل مرئي |
| Terminal | الدخول إلى terminal داخل أي حاوية مباشرة من المتصفح |
| إدارة Volumes | عرض وحذف وفحص محتوى الـ volumes |
| إدارة Networks | عرض الشبكات الداخلية والخارجية وعلاقات الحاويات |
| تحديث الحاويات | سحب صورة جديدة وإعادة تشغيل الحاوية |
| Docker Compose Stack | عرض الـ stack الكامل ومكوناته |
| Environment Variables | عرض متغيرات البيئة (القيم مخفية للأمان) |

---

#### إعداد Portainer في NPM (Proxy Host)

```
Domain Names:     portainer.yourdomain.com
Forward Hostname: portainer
Forward Port:     9000
Block Common Exploits: ON
Websockets Support: ON  ← إجباري للـ terminal داخل Portainer
SSL Certificate:  Request new (Let's Encrypt)
Force SSL: ON
Access List:      IP Whitelist للمدير فقط (موصى به بشدة)
```

**تحذير أمني مهم:**
Portainer لها وصول كامل لـ Docker socket، وهذا يعني أن من يصل إليها يستطيع فعلياً التحكم الكامل بالـ VPS. لذلك يجب تطبيق الحماية التالية:

```
الحماية 1: Access List في NPM (أقوى وسيلة)
  → السماح فقط لـ IP المدير المعروف

الحماية 2: كلمة مرور Portainer الداخلية
  → غير كلمة المرور الافتراضية عند أول دخول

الحماية 3: Cloudflare Zero Trust (اختياري — أعلى مستوى)
  → مصادقة إضافية قبل الوصول للـ subdomain
```

```
3. كل connection يجب أن يرتبط بـ nodes موجودة بالضبط
4. أضف Error Handler لكل node يمكن أن يفشل (HTTP، Database، Email)
5. سمّي الـ nodes بأسماء وصفية عربية أو إنجليزية
6. استخدم فقط الـ nodes الموجودة في n8n (لا تخترع nodes)

بنية JSON الإجبارية:
{
  "name": "اسم الـ Workflow",
  "active": false,
  "nodes": [...],
  "connections": {...},
  "settings": {"executionOrder": "v1"}

قواعد إضافية:
- للمجدولات اليومية: استخدم scheduleTrigger مع interval hours=24
- لكل HTTP Request: أضف error handler

- connections: المفتاح هو اسم الـ source node، ليس id
```
