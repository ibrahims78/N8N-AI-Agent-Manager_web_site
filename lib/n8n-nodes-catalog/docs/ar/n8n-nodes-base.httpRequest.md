# HTTP Request node

تُعد HTTP Request node إحدى أكثر العقد مرونة في n8n. تسمح لك بإجراء طلبات HTTP للاستعلام عن البيانات من أي تطبيق أو خدمة تحتوي على واجهة برمجة تطبيقات (API) من نوع REST. يمكنك استخدام HTTP Request node كعقدة عادية أو ربطها بـ [وكيل ذكاء اصطناعي (AI agent)](/integrations/builtin/cluster-nodes/root-nodes/n8n-nodes-langchain.agent/tools-agent.md) لاستخدامها كـ [أداة (tool)](/advanced-ai/examples/understand-tools.md){ data-preview }.

عند استخدام هذه العقدة، فإنك تقوم بإنشاء استدعاء REST API. تحتاج إلى فهم أساسي لمصطلحات ومفاهيم API.

هناك طريقتان لإنشاء طلب HTTP: تهيئة [معاملات العقدة](#node-parameters) أو [استيراد أمر curl](#import-curl-command).

> **بيانات الاعتماد**
>
> ارجع إلى [بيانات اعتماد HTTP Request](/integrations/builtin/credentials/httprequest.md) للحصول على إرشادات حول إعداد المصادقة.

## معاملات العقدة

### الطريقة

حدد الطريقة المراد استخدامها للطلب:

- DELETE
- GET
- HEAD
- OPTIONS
- PATCH
- POST
- PUT

### عنوان URL

أدخل نقطة النهاية (endpoint) التي ترغب في استخدامها.

### المصادقة

توصي n8n باستخدام خيار **نوع بيانات الاعتماد المُحددة مسبقًا (Predefined Credential Type)** عندما يكون متاحًا. إنه يوفر طريقة أسهل لإعداد وإدارة بيانات الاعتماد، مقارنة بتهيئة بيانات الاعتماد العامة.

#### بيانات الاعتماد المُحددة مسبقًا

بيانات الاعتماد لعمليات التكامل المدعومة من n8n، بما في ذلك العقد المدمجة والمجتمعية. استخدم **نوع بيانات الاعتماد المُحددة مسبقًا (Predefined Credential Type)** للعمليات المخصصة بدون إعداد إضافي. ارجع إلى [عمليات API المخصصة](/integrations/custom-operations.md) لمزيد من المعلومات.

#### بيانات الاعتماد العامة

بيانات الاعتماد لعمليات التكامل غير المدعومة من n8n. ستحتاج إلى تهيئة عملية المصادقة يدويًا، بما في ذلك تحديد نقاط نهاية API المطلوبة، والمعاملات الضرورية، وطريقة المصادقة.

يمكنك تحديد إحدى الطرق التالية:

* Basic auth
* Custom auth
* Digest auth
* Header auth
* OAuth1 API
* OAuth2 API
* Query auth

ارجع إلى [بيانات اعتماد HTTP request](/integrations/builtin/credentials/httprequest.md) لمزيد من المعلومات حول إعداد كل نوع من أنواع بيانات الاعتماد.

### إرسال معاملات الاستعلام

تعمل معاملات الاستعلام كمرشحات على طلبات HTTP. إذا كانت واجهة برمجة التطبيقات (API) التي تتفاعل معها تدعمها وكان الطلب الذي تجريه يحتاج إلى مرشح، فقم بتشغيل هذا الخيار.

**حدد معاملات الاستعلام** الخاصة بك باستخدام أحد الخيارات المتاحة:

*   **باستخدام الحقول أدناه (Using Fields Below)**: أدخل أزواج **الاسم (Name)**/**القيمة (Value)** لمعاملات **الاستعلام (Query Parameters)**. لإدخال المزيد من أزواج اسم/قيمة معامل الاستعلام، حدد **إضافة معامل (Add Parameter)**. الاسم هو اسم الحقل الذي تقوم بالتصفية عليه، والقيمة هي قيمة المرشح.
*   **باستخدام JSON (Using JSON)**: أدخل **JSON** لتعريف معاملات الاستعلام الخاصة بك.

ارجع إلى وثائق API لخدمتك للحصول على إرشادات مفصلة.

### إرسال الرؤوس

استخدم هذا المعامل لإرسال رؤوس مع طلبك. تحتوي الرؤوس على بيانات وصفية أو سياق حول طلبك.

**حدد الرؤوس** باستخدام أحد الخيارات المتاحة:

*   **باستخدام الحقول أدناه**: أدخل أزواج **الاسم**/**القيمة** لـ **معاملات الرأس**. لإدخال المزيد من أزواج الاسم/القيمة لمعاملات الرأس، حدد **إضافة معامل**. الاسم هو الرأس الذي ترغب في تعيينه، والقيمة هي القيمة التي تريد تمريرها لهذا الرأس.
*   **باستخدام JSON**: أدخل **JSON** لتعريف معاملات الرأس الخاصة بك.

ارجع إلى توثيق API الخاص بخدمتك للحصول على إرشادات مفصلة.

### إرسال النص

إذا كنت بحاجة إلى إرسال نص مع طلب API الخاص بك، فقم بتشغيل هذا الخيار.

ثم حدد **نوع محتوى النص** الذي يتطابق بشكل أفضل مع تنسيق محتوى النص الذي ترغب في إرساله.

<!-- vale Vale.Spelling = NO -->
#### Form URLencoded
<!-- vale Vale.Spelling = YES -->

استخدم هذا الخيار لإرسال نصك بتنسيق `application/x-www-form-urlencoded`.

**حدد النص** باستخدام أحد الخيارات المتاحة:

*   **باستخدام الحقول أدناه**: أدخل أزواج **الاسم**/**القيمة** لـ **معاملات النص**. لإدخال المزيد من أزواج الاسم/القيمة لمعاملات النص، حدد **إضافة معامل**. يجب أن يكون الاسم هو اسم حقل النموذج، والقيمة هي ما ترغب في تعيين هذا الحقل إليه.
*   **باستخدام حقل واحد**: أدخل أزواج الاسم/القيمة الخاصة بك في معامل **نص** واحد بالتنسيق `fieldname1=value1&fieldname2=value2`.

ارجع إلى توثيق API الخاص بخدمتك للحصول على إرشادات مفصلة.

#### Form-Data

استخدم هذا الخيار لإرسال نصك بتنسيق `multipart/form-data`.

قم بتكوين **معاملات النص** الخاصة بك عن طريق تحديد **نوع المعامل**:

*   اختر **بيانات النموذج** لإدخال أزواج **الاسم**/**القيمة**.
*   اختر **ملف n8n الثنائي** لسحب النص من ملف يمكن للعقدة الوصول إليه.
    *   **الاسم**: أدخل معرف الحقل المراد تعيينه.
    *   **اسم حقل بيانات المدخلات**: أدخل اسم الحقل الوارد الذي يحتوي على بيانات الملف الثنائي التي تريد معالجتها.

حدد **إضافة معامل** لإدخال المزيد من المعاملات.

ارجع إلى توثيق API الخاص بخدمتك للحصول على إرشادات مفصلة.

#### JSON

استخدم هذا الخيار لإرسال نصك بتنسيق JSON.

**حدد النص** باستخدام أحد الخيارات المتاحة:

*   **باستخدام الحقول أدناه**: أدخل أزواج **الاسم**/**القيمة** لـ **معاملات النص**. لإدخال المزيد من أزواج الاسم/القيمة لمعاملات النص، حدد **إضافة معامل**.
*   **باستخدام JSON**: أدخل **JSON** لتعريف نصك.

ارجع إلى توثيق API الخاص بخدمتك للحصول على إرشادات مفصلة.

#### ملف ثنائي في n8n

استخدم هذا الخيار لإرسال محتويات ملف مخزن في n8n كجسم للطلب.

أدخل اسم الحقل الوارد الذي يحتوي على الملف كـ **Input Data Field Name**.

ارجع إلى توثيق API الخاص بخدمتك للحصول على إرشادات مفصلة حول كيفية تنسيق الملف.

#### خام

استخدم هذا الخيار لإرسال بيانات خام في جسم الطلب.

*   **Content Type**: أدخل ترويسة `Content-Type` لاستخدامها لمحتوى الجسم الخام. ارجع إلى توثيق IANA [أنواع الوسائط](https://www.iana.org/assignments/media-types/media-types.xhtml) للحصول على قائمة كاملة بأنواع محتوى MIME.
*   **Body**: أدخل محتوى الجسم الخام المراد إرساله.

ارجع إلى توثيق API الخاص بخدمتك للحصول على إرشادات مفصلة.

## خيارات العقدة

حدد **Add Option** لعرض هذه الخيارات وتحديدها. الخيارات متاحة لجميع المعاملات ما لم يُذكر خلاف ذلك.

### تنسيق المصفوفة في معاملات الاستعلام

> **توفر الخيار**
>
> هذا الخيار متاح فقط عند تشغيل **Send Query Parameters**.

استخدم هذا الخيار للتحكم في تنسيق المصفوفات المضمنة في معاملات الاستعلام. اختر من هذه الخيارات:

*   **بدون أقواس**: سيتم تنسيق المصفوفات كـ name=value لكل عنصر في المصفوفة، على سبيل المثال: `foo=bar&foo=qux`.
*   **أقواس فقط**: تضيف العقدة أقواسًا مربعة بعد كل اسم مصفوفة، على سبيل المثال: `foo[]=bar&foo[]=qux`.
*   **أقواس مع فهارس**: تضيف العقدة أقواسًا مربعة بقيمة فهرس بعد كل اسم مصفوفة، على سبيل المثال: `foo[0]=bar&foo[1]=qux`.

ارجع إلى توثيق API الخاص بخدمتك للحصول على إرشادات حول الخيار الذي يجب استخدامه.

### التجميع على دفعات

تحكم في كيفية تجميع أعداد كبيرة من عناصر الإدخال على دفعات:

*   **العناصر لكل دفعة**: أدخل عدد عناصر الإدخال لتضمينها في كل دفعة.
*   **الفاصل الزمني للدفعة**: أدخل الوقت للانتظار بين كل دفعة من الطلبات بالمللي ثانية. أدخل 0 لعدم وجود فاصل زمني للدفعة.

### تجاهل مشكلات SSL

بشكل افتراضي، يقوم n8n بتنزيل الاستجابة فقط إذا نجح التحقق من شهادة SSL. إذا كنت ترغب في تنزيل الاستجابة حتى إذا فشل التحقق من شهادة SSL، فقم بتشغيل هذا الخيار.

### ترويسات بأحرف صغيرة

اختر ما إذا كنت تريد تحويل أسماء الترويسات إلى أحرف صغيرة (مفعل، افتراضي) أم لا (معطل).

### عمليات إعادة التوجيه

اختر ما إذا كنت تريد اتباع عمليات إعادة التوجيه (مفعلة افتراضيًا) أم لا (معطلة). إذا كانت مفعلة، أدخل العدد الأقصى لعمليات إعادة التوجيه التي يجب أن يتبعها الطلب في **Max Redirects**.

### الاستجابة

استخدم هذا الخيار لتعيين بعض التفاصيل حول استجابة API المتوقعة، بما في ذلك:

*   **تضمين رؤوس الاستجابة والحالة**: افتراضيًا، تُرجع العقدة (Node) الجسم فقط. قم بتفعيل هذا الخيار لإرجاع الاستجابة الكاملة (الرؤوس ورمز حالة الاستجابة) بالإضافة إلى الجسم.
*   **عدم إرجاع خطأ أبدًا**: افتراضيًا، تُرجع العقدة (Node) النجاح فقط عندما تعود الاستجابة برمز 2xx. قم بتفعيل هذا الخيار لإرجاع النجاح بغض النظر عن الرمز المُعاد.
*   **تنسيق الاستجابة**: حدد التنسيق الذي يتم به إرجاع البيانات. اختر من:
    *   **اكتشاف تلقائي** (افتراضي): تكتشف العقدة (Node) الاستجابة وتنسقها بناءً على البيانات المُعادة.
    *   **ملف**: حدد هذا الخيار لوضع الاستجابة في ملف. أدخل اسم الحقل الذي تريد إرجاع الملف فيه في **Put Output in Field**.
    *   **JSON**: حدد هذا الخيار لتنسيق الاستجابة كـ JSON.
    *   **نص**: حدد هذا الخيار لتنسيق الاستجابة كنص عادي. أدخل اسم الحقل الذي تريد إرجاع الملف فيه في **Put Output in Field**.

### تقسيم الصفحات

استخدم هذا الخيار لتقسيم النتائج إلى صفحات، وهو مفيد للتعامل مع نتائج الاستعلام (Query) التي تكون كبيرة جدًا بحيث لا يمكن لـ API إرجاعها في استدعاء واحد.

> **افحص بيانات API أولاً**
>
> تتطلب بعض خيارات تقسيم الصفحات معرفة بالبيانات التي تُرجعها واجهة API التي تستخدمها. قبل إعداد تقسيم الصفحات، إما أن تتحقق من وثائق API، أو تقوم باستدعاء API بدون تقسيم الصفحات، لرؤية البيانات التي تُرجعها.
??? Details "فهم تقسيم الصفحات"
    يعني تقسيم الصفحات تقسيم مجموعة كبيرة من البيانات إلى صفحات متعددة. تعتمد كمية البيانات في كل صفحة على الحد الذي تحدده.
  
    على سبيل المثال، تقوم باستدعاء API إلى نقطة نهاية تسمى `/users`. تريد API إرسال معلومات عن 300 مستخدم، ولكن هذه كمية كبيرة جدًا من البيانات لترسلها API في استجابة واحدة.
  
    إذا كانت API تدعم تقسيم الصفحات، يمكنك جلب البيانات بشكل تدريجي. للقيام بذلك، تستدعي `/users` بحد لتقسيم الصفحات، ورقم صفحة أو URL لإخبار API بالصفحة التي يجب إرسالها. في هذا المثال، لنفترض أنك تستخدم حدًا قدره 10، وتبدأ من الصفحة 0. ترسل API أول 10 مستخدمين في استجابتها. ثم تستدعي API مرة أخرى، وتزيد رقم الصفحة بمقدار 1، للحصول على النتائج العشرة التالية.

قم بتكوين إعدادات تقسيم الصفحات:

*   **وضع تقسيم الصفحات**:
    *   **إيقاف**: إيقاف تقسيم الصفحات.
    *   **تحديث معامل (Parameter) في كل طلب**: استخدم هذا عندما تحتاج إلى تعيين المعاملات (Parameters) ديناميكيًا لكل طلب.
    *   **الاستجابة تحتوي على URL الصفحة التالية**: استخدم هذا عندما تتضمن استجابة API عنوان URL للصفحة التالية. استخدم تعبيرًا (Expression) لتعيين **Next URL**.

لإعدادات الأمثلة، ارجع إلى [دليل عقدة طلب HTTP | تقسيم الصفحات](/code/cookbook/http-node/pagination.md).

توفر n8n متغيرات مدمجة للعمل مع طلبات واستجابات عقدة HTTP عند استخدام تقسيم الصفحات:

| المتغير | الوصف |
| -------- | ----------- |
| `$pageCount` | عدد صفحات تقسيم الصفحات. يتتبع عدد الصفحات التي جلبتها العقدة (Node). |
| `$request` | كائن الطلب المُرسل بواسطة عقدة HTTP. |
| `$response` | كائن الاستجابة من استدعاء HTTP. يتضمن `$response.body` و`$response.headers` و`$response.statusCode`. تعتمد محتويات `body` و`headers` على البيانات المُرسلة بواسطة API. |

> **اختلافات API**
>
> تطبق واجهات API المختلفة تقسيم الصفحات بطرق مختلفة. تحقق من وثائق API لواجهة API التي تستخدمها للحصول على التفاصيل. تحتاج إلى معرفة أمور مثل:

* هل توفر واجهة برمجة التطبيقات (API) عنوان URL للصفحة التالية؟
* هل توجد قيود خاصة بواجهة برمجة التطبيقات على حجم الصفحة أو رقم الصفحة؟
* بنية البيانات التي تُرجعها واجهة برمجة التطبيقات.

### Proxy

استخدم هذا الخيار إذا كنت بحاجة إلى تحديد وكيل HTTP.

أدخل **Proxy** الذي يجب أن يستخدمه الطلب. هذا له الأسبقية على الإعدادات العامة المعرفة باستخدام [متغيرات البيئة `HTTP_PROXY` أو `HTTPS_PROXY` أو `ALL_PROXY`](/hosting/configuration/environment-variables/deployment.md).

### Timeout

استخدم هذا الخيار لتحديد المدة التي يجب أن تنتظرها العقدة حتى يرسل الخادم رؤوس الاستجابة (ويبدأ نص الاستجابة). تُجهض العقدة الطلبات التي تتجاوز هذه القيمة للاستجابة الأولية.

أدخل وقت **مهلة الانتهاء** بالمللي ثانية.

## خيارات خاصة بالأداة فقط

الخيارات التالية متاحة فقط عند إرفاقها [بوكيل ذكاء اصطناعي](/integrations/builtin/cluster-nodes/root-nodes/n8n-nodes-langchain.agent/tools-agent.md) [كأداة](/advanced-ai/examples/understand-tools.md){ data-preview }.

### Optimize Response

ما إذا كان سيتم تحسين استجابة الأداة لتقليل كمية البيانات التي يتم تمريرها إلى نموذج اللغة الكبير (LLM). يمكن أن يؤدي تحسين الاستجابة إلى تقليل التكاليف ويمكن أن يساعد نموذج اللغة الكبير على تجاهل التفاصيل غير المهمة، مما يؤدي غالبًا إلى نتائج أفضل.

عند تحسين الاستجابات، يمكنك تحديد نوع استجابة متوقع، والذي يحدد الخيارات الأخرى التي يمكنك تهيئتها. أنواع الاستجابات المدعومة هي:

#### JSON

عند توقع استجابة **JSON**، يمكنك تهيئة الأجزاء التي سيتم استخدامها من بيانات JSON كاستجابة بالخيارات التالية:

*   **Field Containing Data**: يحدد هذا الحقل جزءًا معينًا من كائن JSON الذي يحتوي على بياناتك ذات الصلة. يمكنك ترك هذا فارغًا لاستخدام الاستجابة بأكملها.
*   **Include Fields**: هذه هي الطريقة التي تختار بها الحقول التي تريدها في كائن الاستجابة الخاص بك. توجد ثلاثة خيارات:
    *   **All**: تضمين جميع الحقول في كائن الاستجابة.
    *   **Selected**: تضمين الحقول المحددة أدناه فقط.
        *   **Fields**: قائمة حقول مفصولة بفاصلات لتضمينها في الاستجابة. يمكنك استخدام تدوين النقطة لتحديد الحقول المتداخلة. يمكنك سحب الحقول من لوحة المدخلات لإضافتها إلى قائمة الحقول.
    *   **Exclude**: تضمين جميع الحقول *باستثناء* الحقول المحددة أدناه.
        *   **Fields**: قائمة حقول مفصولة بفاصلات لاستبعادها من الاستجابة. يمكنك استخدام تدوين النقطة لتحديد الحقول المتداخلة. يمكنك سحب الحقول من لوحة المدخلات لإضافتها إلى قائمة الحقول.

#### HTML

عند توقع **HTML**، يمكنك تحديد الجزء ذي الصلة من مستند HTML لنموذج اللغة الكبير (LLM) وتحسين الاستجابة باستخدام الخيارات التالية:

*   **المُحدِّد (CSS)**: عنصر محدد أو نوع عنصر لتضمينه في استجابة HTML. يستخدم العنصر `body` افتراضيًا.
*   **إرجاع المحتوى فقط**: ما إذا كان سيتم تجريد علامات HTML وسماتها من الاستجابة، مع ترك المحتوى الفعلي فقط. يستخدم هذا عددًا أقل من الرموز وقد يكون فهمه أسهل للنموذج.
    *   **العناصر المراد حذفها**: قائمة محددات CSS مفصولة بفواصل لاستبعادها عند استخراج المحتوى.
*   **اقتطاع الاستجابة**: ما إذا كان سيتم تحديد حجم الاستجابة لتوفير الرموز.
    *   **الحد الأقصى لأحرف الاستجابة**: الحد الأقصى لعدد الأحرف المراد تضمينها في استجابة HTML. القيمة الافتراضية هي 1000.

#### Text

عند توقع استجابة نصية عامة (**Text**)، يمكنك تحسين النتائج باستخدام الخيارات التالية:

*   **اقتطاع الاستجابة**: ما إذا كان سيتم تحديد حجم الاستجابة لتوفير الرموز.
    *   **الحد الأقصى لأحرف الاستجابة**: الحد الأقصى لعدد الأحرف المراد تضمينها في استجابة HTML. القيمة الافتراضية هي 1000.

## استيراد أمر curl

[curl](https://curl.se/) هي أداة سطر أوامر ومكتبة لنقل البيانات باستخدام عناوين URL.

يمكنك استخدام curl لاستدعاء واجهات برمجة تطبيقات REST. إذا كانت وثائق API للخدمة التي ترغب في استخدامها توفر أمثلة curl، يمكنك نسخها من الوثائق ولصقها في n8n لتكوين عقدة HTTP Request.

استيراد أمر curl:

> **تنسيق الاستيراد**
>
> يقوم هذا الخيار دائمًا باستيراد أي قيم للمعاملات كسلاسل نصية. إذا كنت ترغب في الحفاظ على نوع الأرقام والقيم المنطقية في طلبك، قم بالتبديل من **Using Fields Below** إلى **Using JSON** والصق كائن JSON الخاص بك الذي يحتوي على المعاملات.

1.  من علامة التبويب **Parameters** الخاصة بعقدة HTTP Request، حدد **Import cURL**. تُفتح نافذة **Import cURL command** المنبثقة.
2.  الصق أمر curl الخاص بك في مربع النص.
3.  حدد **Import**. يقوم n8n بتحميل تكوين الطلب في حقول العقدة. يؤدي هذا إلى الكتابة فوق أي تكوين موجود.

# مشكلات شائعة في عقدة طلب HTTP (HTTP Request node)

فيما يلي بعض الأخطاء والمشكلات الشائعة المتعلقة بـ [عقدة طلب HTTP (HTTP Request node)](/integrations/builtin/core-nodes/n8n-nodes-base.httprequest/index.md) وخطوات حلها أو استكشاف أخطائها وإصلاحها.

## طلب سيء - يرجى التحقق من المعاملات الخاصة بك

يظهر هذا الخطأ عندما تتلقى العقدة خطأ 400 يشير إلى طلب سيء. يحدث هذا الخطأ غالبًا للأسباب التالية:

*   تستخدم اسمًا أو قيمة غير صالحة في **معامل استعلام (Query Parameter)**.
*   تمرر قيم مصفوفة في **معامل استعلام (Query Parameter)** ولكن المصفوفة ليست منسقة بشكل صحيح. حاول استخدام خيار [**تنسيق المصفوفة في معاملات الاستعلام (Array Format in Query Parameters)**](/integrations/builtin/core-nodes/n8n-nodes-base.httprequest/index.md#array-format-in-query-parameters).

راجع توثيق API لخدمتك لتنسيق معاملات الاستعلام الخاصة بك.

## المورد الذي تطلبه لا يمكن العثور عليه

يظهر هذا الخطأ عندما يكون **عنوان URL** لنقطة النهاية الذي أدخلته غير صالح.

قد يكون هذا بسبب خطأ إملائي في عنوان URL أو استخدام API مهملة. ارجع إلى توثيق API لخدمتك للتحقق من أن لديك نقطة نهاية صالحة.

## يجب أن يكون معامل JSON بتنسيق JSON صالح

يظهر هذا الخطأ عندما تكون قد مررت معاملًا بتنسيق JSON وهو غير منسق كـ JSON صالح.

لحل المشكلة، راجع JSON الذي أدخلته بحثًا عن هذه المشكلات:

*   اختبر JSON الخاص بك في مدقق JSON أو محلل بناء الجملة للعثور على أخطاء مثل علامات الاقتباس المفقودة، الفواصل الزائدة أو المفقودة، المصفوفات المنسقة بشكل غير صحيح، الأقواس المربعة أو المتعرجة الزائدة أو المفقودة، وما إلى ذلك.
*   إذا كنت قد استخدمت **تعبيرًا (Expression)** في العقدة، فتأكد من أنك قمت بتضمين JSON بالكامل بين أقواس متعرجة مزدوجة، على سبيل المثال:
    ```
    {{
        {
        "myjson":
        {
            "name1": "value1",
            "name2": "value2",
            "array1":
                ["value1","value2"]
        }
        }
    }}
    ```

## ممنوع - ربما تحقق من بيانات الاعتماد الخاصة بك

يظهر هذا الخطأ عندما تتلقى العقدة خطأ 403 يشير إلى فشل المصادقة (Authentication).

لحل المشكلة، راجع بيانات الاعتماد المحددة وتأكد من إمكانية المصادقة بها. قد تحتاج إلى:

*   تحديث الأذونات أو النطاقات بحيث يمكن لمفتاح API الخاص بك أو حسابك تنفيذ العملية (Operation) التي حددتها.
*   تنسيق بيانات الاعتماد العامة الخاصة بك بطريقة مختلفة.
*   إنشاء مفتاح API جديد أو رمز مميز (token) بأذونات أو نطاقات مناسبة.

## 429 - الخدمة تتلقى عددًا كبيرًا جدًا من الطلبات منك

يظهر هذا الخطأ عندما تتلقى العقدة (Node) [خطأ 429](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status/429) من الخدمة التي تستدعيها. غالبًا ما يعني هذا أنك تجاوزت حدود المعدل لتلك الخدمة. يمكنك معرفة المزيد في صفحة [التعامل مع حدود معدل API](/integrations/builtin/rate-limits.md).

لحل الخطأ، يمكنك استخدام أحد الخيارات المضمنة في عقدة طلب HTTP (HTTP Request):

### التجميع في دفعات

استخدم هذا الخيار لإرسال الطلبات في دفعات (Batches) وإدخال تأخير بينها.

1.  في عقدة طلب HTTP، حدد **إضافة خيار > التجميع في دفعات**.
2.  عيّن **العناصر لكل دفعة (Items per Batch)** إلى عدد عناصر المدخلات (Input Items) التي سيتم تضمينها في كل طلب.
3.  عيّن **الفاصل الزمني للدفعة (مللي ثانية) (Batch Interval (ms))** لإدخال تأخير بين الطلبات بالمللي ثانية. على سبيل المثال، لإرسال طلب واحد إلى واجهة برمجة تطبيقات (API) في الثانية، عيّن **الفاصل الزمني للدفعة (مللي ثانية)** إلى `1000`.

### إعادة المحاولة عند الفشل

استخدم هذا الخيار لإعادة محاولة العقدة بعد محاولة فاشلة.

1.  في عقدة طلب HTTP، انتقل إلى **الإعدادات (Settings)** ومكّن **إعادة المحاولة عند الفشل (Retry on Fail)**.
2.  عيّن **الحد الأقصى للمحاولات (Max Tries)** إلى الحد الأقصى لعدد المرات التي يجب أن تعيد فيها n8n محاولة العقدة.
3.  عيّن **الانتظار بين المحاولات (مللي ثانية) (Wait Between Tries (ms))** إلى التأخير المطلوب بالمللي ثانية بين عمليات إعادة المحاولة. على سبيل المثال، للانتظار لمدة ثانية واحدة قبل إعادة محاولة الطلب مرة أخرى، عيّن **الانتظار بين المحاولات (مللي ثانية)** إلى `1000`.
